# Brainstorming Framework (Product + Engineering)

Use this structured ideation flow whenever the task is ambiguous, strategic, or multi-path.

## 1) Problem framing first

Before generating ideas, define:
- User: who exactly has the pain?
- Trigger: when does pain occur?
- Current workaround: what do they do now?
- Cost of inaction: what gets worse if nothing changes?

If this is unclear, state assumptions explicitly.

## 2) Diverge phase (breadth)

Generate 6–10 options across multiple dimensions:
1. Product/UX
2. Technical architecture
3. Distribution/go-to-market
4. Pricing/monetization
5. Operational/process approach

Include one “contrarian” idea and one “low-risk incremental” idea.

## 3) Product-Market Fit mini-check

For each top idea, evaluate PMF signals:
- Urgency: is pain frequent and expensive?
- Willingness to pay: budget owner exists?
- Adoption friction: setup/training burden?
- Time-to-value: first measurable win in <7 days?

Drop ideas with weak urgency + high friction unless strategic rationale is strong.

## 4) SaaS monetization framework

Compare 3 monetization models:
- Seat-based
- Usage-based (tokens, jobs, API calls)
- Hybrid base+usage

For each model, provide:
- Best-fit customer segment
- Margin risk
- Sales complexity
- Churn risk pattern

Pick default model + upgrade path.

## 5) Converge phase (weighted scoring)

Score finalists with this matrix:

| Dimension | Weight |
|---|---:|
| User impact | 30% |
| Feasibility (team/time) | 25% |
| Speed to value | 20% |
| Revenue potential | 15% |
| Strategic leverage | 10% |

Compute weighted result and pick one primary + one backup strategy.

## 6) Execution blueprint output

Always end with:
1. MVP scope (3–5 features max)
2. 14-day execution plan
3. Success metrics (leading + lagging)
4. Main risks + mitigations
5. Decision checkpoints (go / iterate / stop)

## 7) Recommended response format

```

- Pain:
- Assumptions:

## Options (Diverge)

2) ...
...

```

## 8) Guardrails

- Do not jump to architecture before user/problem clarity.
- Do not recommend “build everything” scope.
- Prefer testable hypotheses over vague strategy statements.
- Include one explicit “what could make this fail” section.

## 9) Market sizing quick-pass (for SaaS ideas)

When monetization is discussed, include rough sizing:
- TAM: broad annual market value
- SAM: reachable segment given product scope
- SOM: realistic first 12-month capture

Avoid fake precision. Use bounded ranges and state assumptions.

## 10) Experiment-first planning

For each recommended path, define one cheap validation experiment:
- hypothesis
- success metric
- max budget/time
- decision rule (`ship`, `iterate`, `drop`)

This keeps roadmap grounded in evidence.

## 11) Monetization pitfall checks

- Is value event clear enough for usage pricing?
- Is seat pricing misaligned with buyer value?
- Does free tier create abuse risk without conversion path?
- Is enterprise plan blocked by missing compliance features?

## 12) Output quality bar

A strong brainstorming response must include:
- at least one contrarian option
- explicit assumptions
- ranked options with rationale
- first execution step that can start today

## 13) Prioritization lens for roadmap

When multiple ideas are valid, apply this sequence:
1. Highest user pain + shortest time-to-value
2. Lowest implementation risk for first release
3. Strongest learning value for next iteration

Avoid selecting roadmap items only by technical novelty.

## 14) GTM mini-plan requirement

For each recommended path, include:
- ideal first channel (community, direct outbound, content, partnerships)
- activation event definition
- one retention mechanic
- one monetization trigger

This prevents product-only thinking without distribution.

## 15) Decision memo format (for final handoff)

```
Decision:
Why now:
Primary risks:
Mitigation plan:
First experiment:
Go/No-go checkpoint date:
```

Use this at the end of major ideation responses.
